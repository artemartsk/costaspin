## Контекст задачи
В рамках задачи по календарю мы успешно отвязали логику `usePractitionerSchedules` от хардкода. Однако файл `src/lib/mock-data.ts` все еще существует, и подавляющее большинство хуков внутри `src/hooks/useData.ts` содержат "двойную логику" (fallback на статику, если `isSupabaseConfigured == false`).

**Почему это плохо:**
Frontend-моки искажают результаты QA-тестов. Компонент может "зеленеть" в автотестах на локальных моках, в то время как боевая БД (Supabase) требует других связей. Настоящее TDD-тестирование должно проверять именно интеграцию с данными.

## Цели (Acceptance Criteria)

### 1. Удаление статики
- [ ] Полностью удалить файл `src/lib/mock-data.ts` из проекта.

### 2. Рефакторинг Data Hooks (`useData.ts`)
- [ ] Пройти по всем функциям в `src/hooks/useData.ts` (Например: `usePatients`, `usePractitioners`, `useAppointments` и т.д.).
- [ ] Удалить блок `if (!isSupabaseConfigured) return MOCK_X;`.
- [ ] Теперь хуки должны строго выполнять запросы через клиент `supabase`. Если Supabase не сконфигурирован — выдавать строгую ошибку (Fail Fast), чтобы приложение даже не собиралось в "глухом" режиме.

### 3. Интеграция с Автотестами (QA Strategy)
Поскольку мы удаляем встроенные в код моки, инфраструктура автотестов (которая раньше на них опиралась) упадет. 
Как это внедрять с учетом автотестов:
- [ ] **Для Unit и Component тестов:** Интегрируйте **MSW (Mock Service Worker)**. Перехватывайте HTTP-запросы от клиентского SDK Supabase на сетевом уровне. Это изолирует UI-компоненты от БД, но сохранит валидацию реального транспортного слоя.
- [ ] **Для E2E тестов (Playwright/Cypress):** Тесты должны стучаться не к реальному продакшену, а к локально запущенному инстансу Supabase (`supabase start`). 
- [ ] **Data Seeding для тестов:** В `github actions` или локальном `pre-test` скрипте должен прогоняться файл `supabase/migrations/008_demo_seed.sql`, который наполняет БД нужными тестовыми сценариями (например, пациенты для Elviria).
