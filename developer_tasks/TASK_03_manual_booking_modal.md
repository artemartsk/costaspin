# 📋 Task: Реализация "Manual Booking Modal" в Календаре

**Status:** TODO  
**Assignee:** Frontend Developer  
**Epic:** Calendar & Scheduling  

## Контекст задачи
В компоненте `Calendar.tsx` уже реализована логика расчета пустых слотов ("gaps") и вызов placeholder-модалки "New Booking". Однако, для полноценной ручной работы администратора клиники нам нужно превратить этот плейсхолдер в интеллектуальную и надежную форму создания записи (Appointment).

## Цели (Acceptance Criteria)

### 1. Выбор пациента (Patient Selection)
- Форма должна содержать поле поиска/выпадающий список существующих пациентов клиники (через хук `usePatients`).
- Рядом должна быть кнопка **"Новый пациент"**, которая раскрывает компактную inline-форму (Имя, Фамилия, Email) и **СТРОГО ОБЯЗАТЕЛЬНОЕ ПОЛЕ Телефон (Phone Number)** для быстрого создания профиля, не уходя с календаря. Если телефон не введен - форма должна блокировать создание (он нужен для AI Onboarding).

### 2. Контекст записи (Service & Practitioners)
- **Умный префил:** Если администратор кликнул на пустой слот на пересечении конкретного Врача и Времени, поля "Врач" и "Время/Дата" в модалке должны быть **уже заполнены**.
- **Выбор услуги:** Выпадающий список Services должен подтягивать только те услуги, которые входят в `skill_tags / sub_specialties` выбранного врача (либо просто позволить выбрать из всех активных `useServices`).
- Длительность записи (Duration) должна автоматически подставляться на основе выбранной Услуги (Service `duration_minutes`), но администратор должен иметь возможность её изменить.

### 3. Защита от коллизий и Валидация
- При выборе кабинета (Room) система должна предлагать только те кабинеты, которые свободны в это время. Опционально: можно подсвечивать несоответствие типа кабинета услуге (например, массаж в кабинете мануальщика).
- **Booking Source:** Под капотом, при формировании payload'а для отправки в Supabase, форма обязана жестко прописывать: `booking_source: 'manual'`.
- Ограничение: Записи, создаваемые вручную, должны триггерить тот же самый хук проверки конфликтов. Если администратор каким-то чудом пытается впихнуть бронь в занятое время — выводить красный toast `Booking Conflict: Practitioner is already booked` и не закрывать форму.

### 4. Интеграция с Backend (Хуки)
- Использовать мутацию `useCreateAppointment` (из `useData.ts`) для сохранения.
- При успешном сохранении модалка должна закрыться `setBookingSlot(null)`, а календарь — перерисовать сетку.
- Если пациент новый, сначала вызываем `useCreatePatient`, получаем его `id`, и тут же вызываем `useCreateAppointment`.
