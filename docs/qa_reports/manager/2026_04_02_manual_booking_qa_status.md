## 📍 Текущий статус разработки (Task QA_04)
Инструмент для ручного бронирования вне сетки календаря (`ManualBookingModal`) частично готов к боевому тестированию. Я написал полноценный пакет автотестов (5 штук) для покрытия всех бизнес-требований. Ниже представлен срез состояния кода.

### ✅ Работает стабильно (Functional Base)
Следующий функционал уже реализован фронтенд-командой и подтвержден автотестами без нареканий:
1. **Happy Path:** Создание приема для **Existing Patient** (данные уходят в БД с маркером `booking_source: 'manual'`).
2. **On-the-fly Patient Creation:** Режим создания с новой вкладки "Создать пациента" корректно сохраняет профиль и привязывает его к записи за одну-две транзакции.
3. **Защита от конфликтов (Client-side):** Если администратор пытается записать двух разных людей к одному врачу на одно время, система выдает всплывающее предупреждение и не засоряет базу.

### 🔴 Требует доработки (Бизнес-ограничения)
Два критических требования пока не реализованы. Автотесты для них на данный момент горят красным (TDD):
1. **Фильтрация услуг по навыкам врача:** Модальное окно пока выводит полный список всех услуг (допустим, *Chiropractic*) даже для докторов, которые делают только *Physiotherapy*.
2. **Учет занятости кабинетов:** Дропдаун кабинетов сейчас выводит их все подряд, даже если в это время комната физически занята другим врачом.

**Вердикт:** Модальное окно отлично справляется с главной функцией (Create API), но нуждается во второй итерации от программистов для интеграции фильтров. Передаю задачу разработке.
