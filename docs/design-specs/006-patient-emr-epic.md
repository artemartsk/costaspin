# 📋 Epic 3: Unified Patient Record (EMR) & GDPR Forms
**Status:** ACTIVE / PLANNING
**Epic:** Patient Profile & Medical Data Ecosystem

## 1. Контекст и проблематика
На данный момент у нас есть базовая страница `PatientDetail.tsx`, которая отображает имя, контактные данные, количество визитов и список прошедших/будущих приемов. Пациенты могут переходить по Magic Link и подписывать электронные формы (`PatientForms.tsx`), подтверждая GDPR и согласие на лечение.

**Проблема:** 
Текущая реализация — это CRM-подход. Для медицинской системы (Electronic Medical Record - EMR) этого недостаточно. Врачам необходимо вести историю болезни (Clinical Notes), клинике необходимо хранить юридически значимые PDF-копии подписанных согласий, а администраторам — видеть всю историю общения ИИ (Vapi/WhatsApp) с конкретным пациенту в едином окне.

## 2. Глобальные цели Эпика
Превратить базовый профиль пациента в полноценную медицинскую карту, удовлетворяющую стандартам безопасности здравоохранения Европы (GDPR Art. 9).

### 2.1. Рефакторинг UI профиля (Tabbed Interface)
`PatientDetail.tsx` перегрузится, если мы добавим туда все новые фичи. Нужно внедрить систему вкладок (Tabs):
- **Overview:** Текущая сводка (Stats, Upcoming/Past Appointments).
- **Clinical Records:** История лечения, диагнозы, заметки врачей.
- **Documents & Forms:** Подписанные анкеты с возможностью выгрузки.
- **Communications:** Единый таймлайн общения (VAPI звонки, WhatsApp сообщения).

### 2.2. Clinical Notes (Клинические записи)
Врач должен иметь возможность открыть прошедший прием и оставить медицинскую запись.
- **Сущность:** `appointment_notes` или текстовые поля прямо в `appointments`.
- **Поля:** `subjective` (жалобы, со слов пациента), `objective` (осмотр), `assessment` (диагноз), `plan` (план лечения) — стандарт SOAP.

### 2.3. Генерация PDF для Digital Forms
Электронная подпись (ввод имени `fullName`) сейчас просто сохраняется в БД `patient_forms` в виде JSON (`signature_data`).
Для проверок контролирующими органами (и в случае споров) клиника должна иметь возможность скачать форму в формате PDF (содержащую текст согласия, IP, таймстемп и введенное имя).

### 2.4. Unified AI Timeline
У нас есть данные `call_logs`, но они живут отдельно. На вкладке Communications мы должны выводить ленту:
- 10:00 - *Incoming Call (Vapi)* -> Текст транскрипции.
- 10:05 - *WhatsApp Message (Twilio)* -> Текст "Please fill your forms".
- 10:15 - *Form Completed*.
Это даст полный контекст администратору о том, как ИИ вел лида.

## 3. Декомпозиция на задачи (Sub-Tasks)

Эпик будет разбит на 4 конкретные задачи для команды разработки:

### 🛠 TASK_11: Patient Profile Tabbed Navigation (UI)
- Перевод `PatientDetail.tsx` на компонент `Tabs` (shadcn/ui).
- Сохранение существующего Layout (Header с Аватаром остается на месте).
- Выделение компонента Appointments в отдельный подключаемый блок `PatientAppointmentsTab.tsx`.

### 🛠 TASK_12: Clinical SOAP Notes (Frontend + DB)
- Добавление новой таблицы `clinical_notes` (или полей в `appointments`).
- UI блок на вкладке Clinical Records: Выбор приема -> заполнение SOAP формы (Subjective, Objective, Assessment, Plan).
- Защита (RBAC): Заметки могут читать и писать только врачи (`practitioner`) или главврачи.

### 🛠 TASK_13: GDPR Forms PDF Generation
- Создание Edge-функции (например, `generate-pdf`), которая берет данные из `patient_forms` и генерирует формальный PDF документ.
- Добавление UI на вкладке Documents: список всех подписанных форм с иконками `[ Download PDF ]`.

### 🛠 TASK_14: AI Communications Timeline
- Создание компонента `PatientTimeline.tsx`.
- Написание комплексного SQL-запроса (или RPC-функции) для сбора в единый отсортированный по времени массив: звонков (`call_logs`), заполнений форм (`patient_forms`), и системных логов бронирования.
