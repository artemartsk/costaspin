# 📝 ТЗ: Редизайн модального окна "New Manual Booking"

**Цель:** Снизить когнитивную нагрузку при создании записи, исправить вводящие в заблуждение UI-элементы и ускорить работу администратора за счет понятной навигации внутри попапа.

## 📐 Текстовые прототипы (Wireframes)

### Состояние 1: Поиск существующего (Default)
```text
================================================================
  New Manual Booking                                       [✕]
  🗓 Thursday, 2 April at 09:00  <-- (Clean text, no input box)
----------------------------------------------------------------

  Patient Section
  ╔══════════════════════════════════════════════════════════╗
  ║ [■ Existing Patient ] [  Create New Patient  ]           ║ <-- Segmented Control (Tabs)
  ╚══════════════════════════════════════════════════════════╝
  
  Patient Name *
  [ 🔍 Search or select existing patient...                ▼ ]

----------------------------------------------------------------
  Appointment Details

  Practitioner *                    Service *
  [ Select...                   ▼ ] [ Select...                ▼ ]

  Duration (min)                    Room *
  [ 60                            ] [ Select...                ▼ ]

----------------------------------------------------------------
                                      [ Cancel ] [ Save Booking ]
================================================================
```

### Состояние 2: Создание нового пациента
При переключении таба форма пациента плавно меняется:
```text
... (верхняя часть)
  Patient Section
  ╔══════════════════════════════════════════════════════════╗
  ║ [  Existing Patient ] [■ Create New Patient  ]           ║ 
  ╚══════════════════════════════════════════════════════════╝
  
  First Name *                      Last Name *
  [ Enter first name              ] [ Enter last name            ]

  Phone Number                      Email (Optional)
  [ +34 ...                       ] [ email@example.com          ]
... (нижняя часть)
```

## 🛠 Задачи для разработчика:

**1. Шапка и время (Context Header):**
- Убрать `border` и `bg-white` у контейнера с датой. Дата "Thursday 2 April at 9:00" — это контекст, а не поле ввода.
- Сделать текст чуть приглушенным (напр. `text-muted-foreground`), можно добавить иконку календаря слева для быстрого считывания.

**2. Блок выбора пациента (Patient Details):**
- **CRITICAL:** Убрать компонент Switch/Toggle. Заменить его на `Segmented Control` или компактные `Tabs` (например, из shadcn/ui). Это дает пользователю прозрачный выбор между двумя режимами без двусмысленности.
- Для режима **Existing** — оставляем текущий селект с поиском.
- Для режима **New** — рендерим мини-форму: `First Name` (req), `Last Name` (req), `Phone`/`Email`. 

**3. Блок деталей записи (Appointment Details):**
- Заменить академичный заголовок "Booking Context" на понятный "Appointment Details".
- Добавить визуального "воздуха" (`margin-top`) перед этой секцией, чтобы отделить её от пациента.
- **UX-логика для Duration:** Если `Service` еще не выбран, возможно стоит показывать прочерк или `0`. При выборе `Service`, автоматически подтягивать его `Duration`, чтобы администратору не приходилось вбивать "60" руками (но оставить поле доступным для редактирования, если нужен оверрайд).

**4. Валидация:**
- Кнопка `[ Save Booking ]` должна быть неактивна (`disabled`) до тех пор, пока форма не пройдет базовую валидацию: выбран/создан пациент И выбраны practitioner/service/room.
